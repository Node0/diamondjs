# The Inversion: A Chronicle of the DiamondJS v1.5.1 → v2.0 Migration

*Five phases, ~269 tests, all green. This is the story of how a single XSS hole became a security model — and of the three moments where a human reviewer's attention was the load-bearing component.*

---

## Prologue: The Door That Opened Onto Itself

Begin where the trouble began. Not at the keyboard, not at the first commit, but at the conceptual fault line that everything else buckled against.

A security PR had landed against v1.5.1. Its job was modest: catch dangerous writes to DOM sinks — `innerHTML`, the `on*` handlers, the places where a string becomes executable. The PR did what such PRs do. It enumerated the known-dangerous. It built a list called `UNSAFE_DOM_SINK_PROPERTIES` and checked every write against it.

And then the ground truth caught it failing.

`outerHTML` — a sink every bit as lethal as `innerHTML`, a property that will happily parse `<img onerror=...>` into a live DOM subtree — was not in `PROPERTY_NAME_MAP`. So a write to it silently no-opped. It would also slip the blocklist, because the blocklist only knew the names someone had remembered to write down. The dangerous-sink set is open-ended. The list was finite. The gap between those two facts is exactly where attackers live.

This is the generic blocklist failure mode, and it is not a bug you fix by adding `outerHTML` to the list. It is a *category* error. A blocklist enumerates known-bad and fails **open** on everything it has not foreseen. Add `outerHTML` and tomorrow's `srcdoc`, next quarter's novel sink, the property nobody has heard of yet — all of them sail through. You cannot patch your way out of a structure that is wrong at the root.

The reasonable response would have been to add the missing entry and move on. What happened instead was an escalation: a single failed assertion became a window into a deeper tension, and the deeper tension demanded a system-wide audit so that *every* DiamondJS surface — every binding, every event, every converter — would gain security-by-default with one audited escape hatch.

That escalation became a 669-line Design Decision Record, folded in a 243-line Amendment A1, and rode on top of nearly twelve thousand lines of supporting analysis committed before a single line of implementation was touched. The pre-work commit `cdc6526` reads, in the diffstat, like a geological survey of the terrain about to be excavated: `12,127 insertions, 1 deletion`. One deletion. Everything else was the act of deciding what to build.

The throughline under all of it, stated in the DDR's own words and never once relaxed across five phases:

> **A token must predict its own behavior to a ~32B model, not merely describe it to a human.**

Hold that sentence. It is the gravitational constant of this entire migration. Every rename, every removal, every hard error that follows falls out of it.

---

## The Frame: A Version Number That Was Lying

Before the security work could begin, a smaller lie had to be corrected.

The refactor had originally been labeled **v1.5.2**. A patch. A trivial increment on a stable line. And it was, in the language the DDR would later apply to `unsafe` and `.one-time`, *a name that failed its own audit*. SemVer is a contract: PATCH means backward-compatible fixes, MINOR means backward-compatible additions, MAJOR means breaking changes. This refactor was *nothing but* breaking changes — `unsafe`→`raw`, `.one-time`→`set`, `.delegate` deleted, converter semantics replaced wholesale, `&` eliminated, `with` removed, `.trigger`→`.calls`.

"v1.5.2" said *stable line, trivial patch* while meaning *breaking foundational overhaul*. The version was lying the same way `unsafe` lied — a technically-placed label triggering exactly the wrong prior. So it was fixed the same way every other name in this migration was fixed: by making it accurate. **2.0.0.** And not arbitrarily — DiamondJS was named the SemVer canary for the whole project fleet, because the thing at the bottom of the dependency stack is the one whose compatibility contract carries the most weight. The deepest dependency gets the honest contract first.

The baseline was confirmed — the codebase really was at v1.5.1, verified against structural markers, which meant the 1.5.1→2.0 diff was a *real* diff against *real* code. Not a spec written against assumptions. The reconnaissance claims — the delegate stub wired to a fallback, the `outerHTML` no-op, the one-time write bypassing `bind()` — were now checkable facts to be verified before cutting against them. The boundary was closed. The rest was implementation.

Five phases. Each one checkpointed into its own commit. Each one green before the next began.

## Phase 1 — The Security Spine

*Commit `fd58a7d`. 923 insertions, 179 deletions, 20 files. The phase where the framework learned to fail closed.*

The first move was the inversion, and it is the single most important architectural decision in the entire migration. Stop enumerating the dangerous. Enumerate the **safe**.

The safe-sink set is small, boring, and spec-derivable: `textContent`, `value`, `className`, `checked`, `disabled`, the numeric scalars, the plain-text descriptors — the properties where a string is just a string and can never become code. The new `packages/compiler/src/security.ts` — 163 lines, born whole in this commit — holds `SAFE_SINKS` and a single pure function, `gateSink()`, that renders a verdict at the one place the compiler sees *every* sink write before runtime paths diverge: codegen.

Watch what the inversion does to the `outerHTML` fossil. Under the blocklist, `outerHTML`'s absence from the list meant *allowed*. Under the allowlist, its absence means *not safe* — which means it requires `rawBind`, emits a declared-raw record, and if the developer forgets, it **fails closed**, treated as raw with a warning that hard-gates the build. The `outerHTML`-class gap becomes impossible *by construction*. Ignorance no longer fails to allowed. Ignorance fails to raw. That is the whole game, and it is won at the level of structure, not enumeration.

Then the deeper closure — the one-time bypass. In v1.5.1, `.set` (née `.one-time`) compiled to a naked `el[prop] = value`, a direct property write that never entered `bind()`. A runtime guard living inside `bind()` could not see it. It was structurally blind to exactly the writes it most needed to watch. The fix was not a better runtime guard. The fix was to recognize that **compile-time is the single choke point** — every sink emission, `set` / `rawSet` / `bind` / `rawBind`, now routes through one centralized helper that cannot emit a property write without passing `gateSink()` first. The bypass is closed not by patching the downstream guard but by moving the gate upstream of the divergence. You cannot slip past a checkpoint that sits before the road forks.

### The renames, and why each one earned its keep

`unsafe` → `raw`. The word `unsafe` is a *verdict* — an accusation. But a `rawSet` to a trusted constant is not wrong at all; it is merely unprocessed. `raw` is a *description*, accurate across the entire range from "fine" to "XSS hole," and it reads as a verb-modifier (`rawSet`, `rawBind`) where `unsafeBind` reads as a confession. Rust's `unsafe` earned its connotation in *its* domain. In an XSS-sink context, `unsafe` reads as "this is the hole" — alarmist, and frequently false.

`.trigger` → `.calls`. This one is pure autoregressive hygiene. `.trigger` describes the wrong direction of causation — it reads as *the framework triggers something*, framework-as-actor, when the truth is the reverse: the event happens, you respond. The runner-up, `.triggers`, was *technically correct* — and rejected precisely because it is correct in a way that collides. A model with jQuery in its weights knows `.trigger()` means *dispatch* an event, the opposite of handling one, and will hallucinate a conflict that isn't there. `click.calls="save()"` reads strict left-to-right — subject, verb, object — with no competing prior in any corpus. A technically-correct name with a competing prior is worse than a plainer name with none. That sentence is the rename philosophy in miniature.

`.delegate`, gone — and not just the command. The reconnaissance had reported it as a *hollow stub*: the command parsed, `DiamondCore.delegate()` existed in `core.ts` with real selector-matching logic, but the generator emitted `DiamondCore.on()` for it and never once called the delegate method. It parsed, produced running code, and that code silently degraded to a plain listener. So the migration deleted the command *and* the orphaned runtime method — a thing that pretended to work being more dangerous than a thing that obviously doesn't.

And the silent `|| 'bind'` fallback in `parseBindingCommand` — killed. In v1.5.1, an unknown or retired command quietly became a two-way bind. In a security release, fail-open is unacceptable at the level of a typo. A `.trigger` in a template must now become a hard, actionable error ("renamed to `.calls`"), not a silent two-way bind to a sink.

### The two-tier stink gate

The open question — the one the DDR worried at — was whether the "stink biscuit" should be an enforced gate or merely legible documentation an agentic loop could silently bypass. The answer was the asymmetric split, and it is genuinely clever:

- **`stink:warn`** — an undeclared off-allowlist write, a latent bug by definition, nobody claimed it. **Hard gate.** Build blocks on count > 0.
- **`stink:declared`** — an intentional `rawBind`, the developer signed for it. **Don't gate. Baseline it.** A snapshot of the declared-raw set, checked into `stink-baseline.json`, sorted for clean diffs.

The insight underneath: for a non-safe sink, the *emitted bytes are identical* whether declared or undeclared. You cannot make `innerHTML` safe with code — only with a declaration. So the gate's job is not to block raw. It is to make raw *un-addable in silence*. A new `rawBind` doesn't break the build — it **changes the baseline file, and that diff lands in code review.** Every new escape hatch becomes a reviewed, signed-off event. This is the precise mechanism that defeats the agentic-loop bypass: the tripwire is not "block raw," it is "raw cannot be added invisibly." `tools/stink-check.ts`, 177 lines, mirrors the existing LOC-budget checker — `--check` to enforce, `--update` to rewrite, wired beside `check-loc` in `prepublishOnly`.

And undergirding all of it, the three-segment raw grammar. parse5 lowercases every attribute name, so the camelCase legibility of `rawBind` is a source-only affordance the compiler never sees — `innerHTML.rawBind.to-view` arrives as `innerhtml.rawbind.to-view`. The parser learned to split on dots into `property.command.qualifier`, normalize the property through `PROPERTY_NAME_MAP`, and carry `{ type, raw }` internally without ever flattening the source surface into the forbidden `rawTo-view` token. Three segments in, a clean boolean out.

Phase 1 landed green. And it landed with a latent flaw nobody had noticed yet — a flaw that would become the first of three moments where the migration's real character showed.

## The First Human Moment: The One-Way Flow That Wasn't

Here is where a chronicle has to slow down, because here is where the collaboration reveals what it actually is.

Phase 1 had wired `from-view` — the inbound, DOM-to-model binding — through `DiamondCore.bind(el, prop, () => expr, (v) => expr = v)`. Two functions: a getter *and* a setter. And `bind()`, as written, *unconditionally* ran the getter as a to-view effect. Which meant `from-view` — a binding whose entire identity is *one direction, inbound* — was silently, structurally **two-way.** The model could push back into the sink.

Read that for what it is. A binding named for one direction permitting the opposite direction. On most properties, a curiosity. On `innerHTML.rawBind.from-view` — a developer's deliberately-constructed inbound-only security boundary, the validated path where a websocket or a sibling component writes to the model — it is a hole. A model write would reach `innerHTML` *unvalidated*, around the exact boundary the developer built `from-view` to enforce.

The reviewer caught it. During Phase 2's review, reading the generated output rather than the spec, someone noticed that the getter was always there — that the construct's name and the construct's behavior had quietly diverged. Not a failing test. Not a thrown exception. A discrepancy between what a name promised and what the code did, visible only to a human who was reading for *meaning* and not just for green checkmarks.

The fix was small and exactly right. `bind()`'s getter became `(() => unknown) | undefined`. The to-view effect now runs **only when a getter is passed.** `from-view` emits `DiamondCore.bind(el, prop, undefined, setter)` — DOM to model, the model never reaching the sink. You can read it in `core.ts` today, the example comment sitting at line 136: *"One-way from-view (DOM → model only): NO getter, so the model never..."* The `undefined` in the third argument slot is the whole correction, and it is the kind of correction that looks like nothing and means everything.

The principle the reviewer distilled from it deserves to be carved somewhere permanent:

> **A one-way-named flow must never permit the opposite flow.**

And the timing is the part that matters most for the arc. This had to be fixed *before* Phase 3, because Phase 3 was about to build `ParseResult` and the entire inbound-validation story directly on top of `from-view`. Had the two-way leak survived into Phase 3, the validated-parse path would have been constructed on a foundation that silently permitted the unvalidated write it existed to prevent. The security correction had to land before the thing that depended on it was built. Catch it now, in the cheap moment, or build a cathedral on a cracked footing.

The consequence rippled cleanly: `from-view` was reclassified as genuinely *inbound*, removed from the outbound `SinkOp` set, no longer outbound-gated. Its risk moved to where its risk actually lives — the runtime smell check of Phase 3. And the `raw` flag on `from-view` was preserved as exactly the inbound escape hatch Phase 3 would consume. One reviewer's attention, and three phases' worth of architecture clicked into correct alignment.

## Phase 2 — The Structural Directives, Built From Scratch

*Part of commit `b46041c`. Structural directives, conditionals, loops — and a removal argued from first principles.*

`if.bind` became bare `if`. The `.bind` suffix had described the RHS *mechanism* and not the DOM *effect* — which is why a 32B model misreads `if.bind` as "if this element is bound" rather than "include this element when the condition holds." Drop the suffix entirely, and the justification is airtight in a way worth pinning: `if` has **no sink**. The condition is a boolean controlling whether a subtree *exists*; it never flows into the DOM as content. No sink means no injection surface means **no raw variant** — `rawIf` would be literally meaningless, since "raw" only ever modified sink writes. And `if` is *always reactive*, and reactive-over-static is free, so there is no `if.set` either. No sink, always reactive: nothing for a command suffix to distinguish. The bare word is not a simplification — it is the *complete* truth about the construct.

The runtime grew the machinery to back it. `DiamondCore.if(anchor, branches)` — first truthy `when()` wins, branches built lazily and cached so a toggle reuses the subtree, with the reactive condition reads deliberately sequenced *before* the branch builds so the master effect tracks the condition's dependencies (the engine nulls `activeEffect` after a nested effect, so reads must precede builds — a sharp little ordering constraint that the working notes flag as load-bearing). `DiamondCore.repeat(anchor, itemsGetter, makeItem)`, keyed by item identity, reusing and reordering nodes via `insertBefore` in document order. And `captureScope` — collecting the teardown registered while building a branch or item, so a removed subtree disposes its own effects. Structural directives that clean up after themselves, which is, the notes dryly observe, *stricter than the component root currently manages.*

`with` — gone outright. Three nails, weakest to strongest. It violates the rule that scope must be visible at the use site (`<div with.bind="user">${name}` makes `${name}` mean something resolvable only by looking *up* the tree to an invisible enclosing `with`). It is named after the most notorious scope-confusion construct in JavaScript — *the name is apt for a thing you don't want.* And, decisively, TypeScript rejects the `with` statement outright and strict-mode throws on it, so the authoring layer won't compile it and the output context won't run it. The one legitimate need it served — deep repeated access — is replaced by a view-model getter: explicit, local, *testable*, visible at the use site. The escape hatch for the one real need is strictly better than the feature.

### Amendment A1: The case against bare `else`

And then the sharpest structural argument of the phase, folded in from Amendment A1: **bare `else` removed. Does not ship in 2.0 or any version after it.**

The surface complaint is grammatical. `if="condition"` is an attribute with a value. `else-if="condition"` is an attribute with a value. `repeat.for="item of items"` is an attribute with a value. `else` is bare, valueless, positional — it breaks the grammar every other directive obeys. But the deep complaint is **positional pairing at arbitrary nesting depth.** Drop three `else` clauses into a nested tree and each one pairs with its `if` by a proximity rule the syntax does not encode. The compiler must scan upward. The developer must scan upward. The LLM authoring the template must track a stack of open `if` scopes while writing *forward*, and verify pairing on readback by reconstructing the sibling chain. A misplaced `else` is syntactically valid and silently wrong — an entire bug class with no compile-time signal.

The DDR names this exactly: *this is the structured-programming argument restated for template DSLs.* Goto is seductive for the same reason bare `else` is — maximum expressiveness, zero locality. Dijkstra's objection was never about power; it was about locality of reasoning. Bare `else` fails the same test, and `else-if="hasError"` passes it, because its condition *is* its locality — a model reading it knows what it means without scanning anywhere. The residual catch-all that `else` used to serve becomes an explicit negated `if`: `if="!isLoading && !hasError && !isReady"`. Which is not a workaround but a strict improvement — it *names the conditions it asserts are simultaneously false*, documents the developer's model of the state space, and breaks loudly when a new state is added and this line is not updated. Bare `else` says "none of the above." The explicit form says "specifically these named conditions were all false." The second is a meaningful assertion. The first is a residual bucket.

The compiler now rejects `<div else>`, `<div if.bind>`, `<div rawIf>`, `<div if.set>` — each with a diagnostic that points forward to `else-if="!cond"` or the future `<switch>`. The removal is forward-looking, not merely restrictive. A door closed with a sign on it telling you which door to use instead.

## Phase 3 — The Converters, the Pipe, and the Import as Registry

*The remainder of commit `b46041c`. 2,232 insertions across 31 files — the largest single phase. `generator.ts` alone grew by 387 lines.*

The thing formerly called "value converters" — an Aurelia resource concept, and a genuine source of confusion — became **template formatting/parsing methods**, one class bundling two static methods, `format` and `parse`. `CurrencyConverter.format(this.amount, 'USD')` and `CurrencyConverter.parse(raw, 'USD')`, the currency code bound once and threaded to both legs so round-trip consistency comes free. Static, not instance — bare calls in the compiled output, no allocation, no "what is this object" question for a small model to puzzle over.

The pipe `|` survived, but not as a framework resource — as a **Unix pipe.** `${value | parseRaw | clamp | formatPercent}` lowers to `formatPercent(clamp(parseRaw(this.value)))`, pure function composition, zero framework concepts, semantics every developer and every 32B model already carries. And the non-obvious payoff, the one that justifies keeping the glyph at all: **the pipe makes the audit cheap.** A bare `formatCurrency(this.price)` buried in an arbitrary expression forces the compiler into real static analysis to find transforms. A pipe makes "this token is a transform" *syntactically true*, so the format/parse pairing check becomes trivial. The ergonomic syntax and the correctness machinery are one decision, not two.

### The capitalization convention

Phase 3 had to reconcile two sections of the DDR that, read together, implied a distinction the spec never stated outright. §5.3 showed `formatPercent(clamp(parseRaw(this.value)))` — bare camelCase calls. §5.4 showed `CurrencyConverter.format(this.amount, 'USD')` — PascalCase static methods. The implementation discovered the latent rule binding them: **PascalCase head = a converter class**, lowered to directional `.format`/`.parse`; **camelCase head = a plain transform function**, called directly. The convention was DDR-consistent but unwritten, and naming it was the move that let one pipe scanner serve both worlds.

That scanner, by the way, is hand-written, not a regex split — a character scanner tracking string state and bracket depth, splitting on `|` only at depth zero and outside strings, skipping `||`, treating the pipe as the lowest-precedence operator. It had to be hand-written because the alternative mangles `CurrencyConverter` into `this.CurrencyConverter` the moment a naive prefixer touches it. Pipe heads are bare imports, emitted verbatim, *never* `this.`-prefixed; only the data leaf and the arguments run through the identifier prefixer. The critical fix is a discipline about which tokens are sacred.

### `ParseResult` — and the second human moment

Parse already had to validate. You cannot convert `"abc"` to a number without deciding it isn't one. The DDR made that implicit obligation explicit, and in doing so handed DiamondJS its client-side validation story for free. The shape, from §5.7:

```ts
interface ParseResult<T> {
  valid: boolean;
  value: T | null;
  raw: string;        // user's in-progress text — never clobbered
  error: string | null;
}
```

And here is the second moment where a human's attention changed the trajectory.

The implementing agent was asked to confirm a detail of the shape — specifically, whether a failed parse should carry `undefined`. A reasonable thing to confirm. The description in front of the agent suggested it. The path of least resistance was to nod and adopt it.

The agent did not nod. It held the spec against the description and flagged the divergence: the DDR §5.7 actually specifies `value: T | null` — *`null`, not `undefined`* — *and* a `raw` field that the description had omitted entirely. That `raw` field is not decoration. It is the mechanism by which a half-typed `"$1,2"` stays in the input while the model stays clean, the thing that makes "keep raw on invalid" possible at all. To have silently adopted `undefined` and dropped `raw` would have been to quietly amputate the feature the shape exists to provide.

This is spec-fidelity as an active discipline rather than a passive one. The agent treated the Design Decision Record as the authority and the conversational description as a possibly-lossy summary of it — and when the two disagreed, it surfaced the disagreement instead of resolving it by convenience. `ParseResult.ok(value, raw)` and `ParseResult.fail(raw, message)` shipped as constructors precisely so nobody hand-assembles the shape and it cannot drift. The thing held DDR-faithful. The validation seam stayed intact for whatever renders field errors later.

The from-view ParseResult codegen then fell into place, and the elegance the working notes flag is real: "keep raw on invalid" is *free*, because from-view's `undefined` getter — the very correction the first human moment had forced — means the model never writes the DOM, so *not writing the model on an invalid parse* leaves the user's text untouched by construction. The first human moment and the second meet here, in a single line of generated code, and reinforce each other.

### §5.6 enforcement, landing in the compiler, following the import

The pairing check is contextual, not universal. Not "every `format*` must have a `parse*`" — that over-constrains `formatFileSize` and `formatRelativeTime`, which have no sane inverse. Enforce only where direction *demands* it: a `format*` on the outbound leg of a two-way binding requires a `parse*` in the same class, or it is a **hard compile error**. That is where corruption lives — where a tolerant formatter is idempotent on its own output, so the demo looks correct while the model silently holds a string where it should hold a number, and the failure surfaces only when something *else* reads the model: a sum, a comparison, a database write. You cannot reliably catch a non-throwing silent corruption at runtime. You catch it at compile time by requiring the parser to exist.

The decision the user made here is the structurally interesting one. §5.5 had killed the `transform_functions/` magic folder on the grounds that *the import graph is the registry* — provenance lives in the `import` statement, not in a convention folder that hides origin the way DI does. So when §5.6's enforcement needed to read a converter class to check for a `static parse`, the question was *where that resolution should run.* The answer — Option A — was to put it **in the compiler**, following the import the way §5.5 promised the compiler would. `compileAndInject()` regex-scans the component source for the converter's `import ... from '<path>'`, resolves the path relative to `options.filePath` (trying `.ts`, `.js`, `/index.*`), reads the module, and checks for `/static\s+parse\b/`. A hard `converter-missing-parse` error if it's absent; a soft `converter-unresolved` info for package specifiers and re-exports it can't follow ("verify manually" — fail-soft, never a crash). The registry promised by §5.5 became the enforcement mechanism of §5.6. The same import statement that documents where a converter comes from is the thing the compiler walks to verify the converter is whole.

And a standalone `.diamond.html` — compiled to a module that only imports `DiamondCore`, with none of the author's imports in scope — now errors if it contains named pipe transforms, because they would be undefined symbols at runtime. Fail-closed on the API surface beats a runtime `ReferenceError`.

The phase closed by giving the converters a home: a **new `@diamondjs/converters` package**, 500-LOC budget added to the checker, shipping Currency (a thin `Intl.NumberFormat` wrapper), Date (`Intl.DateTimeFormat`, with a `parseDate` that does calendar-validity checking and *rejects Feb 30* — the highest-value battery because it's the one most developers get wrong), and Phone (NA-only, deliberately simple, with the limitation marked as the teaching example rather than hidden). `ParseResult` stayed in `@diamondjs/runtime` so batteries and user converters import the same contract and it cannot drift. The import graph is the registry, and now the registry has a reference shelf.

## Phase 4 — Timing, and the Glyph That Had to Die

*Commit `0392776`. 384 insertions. The smallest phase, and the one that closed the security argument's last loophole.*

Aurelia's `&` — binding behaviors — was eliminated entirely. Not relocated. *Eliminated.* The DDR's inventory walked every member to its new home: `oneTime` to `set`, the mode behaviors to command suffixes already in place, `signal` to *make the dependency reactive instead*, `updateTrigger` to a new `update-on` attribute, `debounce`/`throttle` to self-registering class methods. No surviving member needed the glyph.

`value.update-on="blur"` — property-scoped, parsed into a per-element map keyed by canonical property, applied to the matching inbound binding as a fifth argument to `bind()`. Property-scoped and not bare for a precise reason: bare `update-on` is ambiguous on a multi-binding element (`<date-range start.two-way="from" end.two-way="to" update-on="blur">` — *which* binding?), so the parser rejects the bare form with a diagnostic and rejects `update-on` on a property that has no inbound binding to time. You can read both guards in `parser.ts` today, at lines 178 and 235.

`this.debounce` / `this.throttle` — protected `Component` methods returning a wrapped function, and the decisive design move is that they **self-register their `cancel` against the component's teardown registry at creation time.** This is the whole reason the sugar earns its existence — not ergonomics, but *owning the cleanup the hand-written version forgets.* The class field `handleInput = this.debounce(v => this.query = v, 500)` is now a genuine one-liner *and* leak-safe, with no visible timer-cancel burden shifted onto the developer — the exact line an LLM otherwise drops because nothing in the source signals an outstanding timer. The field-init ordering even works out: the base class's `cleanups = []` runs during `super()`, the subclass field initializer runs after, so `registerCleanup` is ready when `this.debounce` reaches for it.

### The Third Human Moment: The Bitwise Adjudication

And then the last of the three moments — the one that is purely a judgment call, with the security of the whole release riding on which way it goes.

A lone `&` in a binding expression — not `&&`, not `||`, a single ampersand — is now a **hard error**, detected by `/(?<!&)&(?!&)/`. The question put to the decision: a bitwise `&` is a *legitimate* JavaScript operator. `this.flags & MASK` is real, valid code. Flagging it looks like a false positive. Shouldn't the bitwise case be allowed — or at least softened to a warning?

The adjudication held the line, and the reasoning is the cleanest statement of the framework's philosophy in the entire migration. A template is a **declarative binding surface, not a computation surface.** There is no legitimate bitwise `&` in a binding for the same reason there is none in a SQL `WHERE` clause or a CSS selector — it is the wrong *kind* of place for that operation to live. The redirect is a view-model getter: `get result() { return this.a & this.b }`. Explicit, local, testable, visible.

But the sharper half of the reasoning is about *why a warning was the wrong instrument.* Syntactically, the removed-behavior `&` and the bitwise `&` are *indistinguishable* — both are a lone ampersand, and no regex can separate them. And that indistinguishability is fine, because **both should leave the template.** A warning would let both pass: the rare legitimate bitwise use (trivially fixed with a getter) *and* the rare illegitimate behavior-`&` sneaking past the loop. And the second is the real risk. The DDR's own words, which the implementation adopted verbatim:

> **Softening a security-adjacent error is how holes open.**

A rare legitimate use that is trivially fixed costs almost nothing to forbid. A rare illegitimate use that slips past a warning costs everything. When the cost of a false positive is "write a one-line getter" and the cost of a false negative is "a hole opens silently," the asymmetry decides. Hard error. The diagnostic message redirects both cases — bitwise to a getter, behavior-`&` to `update-on` / `debounce` / a reactive dependency / `set` — so the developer who hits it, in either case, is handed the door they should have used.

This is the same instinct that ran through all three human moments, stated three ways. *A one-way flow must never permit the opposite flow. Hold the spec against the convenient summary. Never soften a security-adjacent error.* Three different surfaces, one disposition: when in doubt, fail closed, and make the closed door point at the open one.

## Phase 5 — The Naming Made True

*The final phase. Versions bumped, the example rewritten end-to-end.*

Every package went to **2.0.0**, cross-dependencies pinned at `^2.0.0`, DiamondJS taking up its role as the SemVer canary for everything built on top of it. The version number stopped lying.

And the example was rewritten not as decoration but as proof. `Tasks.diamond.html` exercises the new binding language whole — `set` and `rawSet`, two-way, `.calls`, bare `if` and `else-if`, `repeat.for`, interpolation. A second component, `MoneyForm`, demonstrates the converter pipe end to end: a `CurrencyConverter` on a two-way binding, `ParseResult` validation, `value.update-on="blur"` so the validating parser fires on commit rather than hostile per-keystroke rejection of `"$1,2"` mid-type, and a debounced side-effect. The example is the migration's own integration test, written in its own new grammar.

And then the single most eloquent artifact of the entire release. The stink baseline, checked into the repository root:

```json
{
  "version": 1,
  "declared": [
    {
      "id": "examples/hello-world/src/Tasks.diamond.html:6:innerHTML:set",
      "property": "innerHTML",
      "op": "set",
      "expression": "bannerHtml"
    }
  ]
}
```

**One** declared raw, in the entire codebase. One audited `innerHTML` write, in the example, signed for, recorded, greppable, and impossible to add a second to without a reviewed diff. The framework that began this migration by *failing open on `outerHTML`* now ends it with a single sanctioned escape hatch and a gate that makes every future one a visible event. That JSON object is what the whole inversion was for.

## Epilogue: One Mechanism Wearing Three Names

Stand back from the five commits and the 269 tests and look at what was actually built, because the DDR's §8 says it better than a summary can: the pipe, the raw path, and the audit are **not three features. They are one mechanism.** The declared, audited XSS escape hatch *is* a sanitizer in a pipe on a raw binding:

```html
innerHTML.rawBind.to-view="userHtml | sanitizeHtml"
```

That single line says: you are writing `innerHTML` (off the allowlist, so forced to `rawBind`, so it fails closed if you forget); you have routed it through a named transform (so the audit can see it syntactically); and the biscuit records both facts as one greppable, baseline-diffable line. The pipe is not in tension with the security work. The pipe is the security work's *declaration surface.* Keep the pipe and the security model in the same hand, and the entire story collapses into one legible record a small model can read without framework knowledge.

The numbers, as material evidence of the effort: the compiler grew to 2,719 lines against a 5,000 budget; the runtime to 483 against 2,500; the new converters package to 84 against 500; the parcel plugin to 213 against 300. **3,499 of 7,800 lines — 45% of the budget — for a security-by-default binding language, a structural-directive system built from scratch, a validating converter framework, and handler-timing primitives that clean up after themselves.** 269 tests, all green. The stink gate green, with its single honest declaration.

But the numbers are not the achievement. The achievement is the disposition the three human moments revealed and the architecture encoded — that a name must predict its behavior, that a one-way flow must never permit its opposite, that you hold the spec against the convenient summary, that you never soften a security-adjacent error because that is how holes open. v1.5.1 enumerated the dangerous and failed open on everything it forgot. v2.0 enumerates the safe and fails closed on everything it doesn't know.

That is not a patch. That is an inversion. And the door that, in the prologue, opened onto itself — now opens outward, onto a surface where ignorance fails to *safe*, and every exception is a signature in a file someone has to read.

*The boundary is closed. The rest is v2.1.*
